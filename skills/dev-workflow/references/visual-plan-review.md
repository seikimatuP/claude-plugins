# Visual Plan-Review Gate (Step 4)

Deep reference for Step 4's **visual gate path**. Read and follow this top to bottom **whenever `visual_plan_review` is enabled** (§ Configuration) and Step 4 has reached sub-step 2 path (b) — do **not** pre-gate reading it on browser reachability. **This reference owns the reachability determination**: step 2 below runs `printenv CLAUDE_CODE_REMOTE` and returns `fallback` when the local browser is unreachable. This file is the single canonical home for the gate procedure; Step 4 delegates here rather than restating it.

Because `visual_plan_review` is enabled, **Step 2 did not enter Plan Mode** (`plan_mode_active = false`; see `SKILL.md` Step 2 sub-step 2's `plan_mode_active` resolution / § Configuration), so this gate runs **outside Plan Mode**: it freely performs the non-read-only operations below (writing the served file, launching `serve.mjs`), there is **no `ExitPlanMode` / approval modal**, and the canonical plan document is `.claude/plans/<slug>.md`, written by Step 4 sub-step 2 path (b) before this procedure runs (step 3 below names the `<slug>` source). Running outside Plan Mode does not reduce plan-creation capability: Step 2 authors the plan in the main thread on both paths, and on this visual path Step 2's "Codebase-research delegation" (see `SKILL.md` Step 2 sub-step 3) reproduces — **when it fires** (it is optional and guard-gated, with an inline-research fallback) — the research-context isolation that Plan Mode's built-in read-only `Plan` subagent provides on the `visual_plan_review: false` path, so the visual path forgoes only Plan Mode's automatic isolation, which the Step 2 delegation restores when it fires (and Step 3 plan review runs identically on both paths as the independent quality gate).

The gate replaces Step 4's text approval presentation with a browser-based, structured plan review: the viewer renders the plan as a scannable review surface (summary header, collapsible sections, Decision cards with a Recommendation/Alternative one-click switch, per-element comment affordances, and mermaid diagrams) rather than raw markdown. It returns one of three outcomes to Step 4: **`approve`**, **`rewrite-approach`**, or **`fallback`** (defined in § Decision mapping and § Fallback contract below).

## serve.mjs contract

The viewer (`scripts/plan-review/serve.mjs`) ships with this skill; this gate only invokes it. The server is **transport only** — it ships the raw plan markdown and collects the submit; all plan-structure parsing and rendering happen browser-side in `public/index.html`. Its contract:

- **Invocation**: `node <serve.mjs> --plan <path> [--prev <path>] --lang <resolved-language> [--timeout <seconds>] --wait` — serves the Markdown plan at `<path>` on a random `127.0.0.1` port, opens the local default browser, and blocks until the user submits. `--lang` is the workflow's resolved `language` (`ja` / `en`); it controls **only** the language of browser-generated text — the "switch to alternative" comment body and the diff banner (all other UI chrome is English) — and defaults to `en` when omitted. `--prev <path>` is the plan version the user reviewed on the previous launch (the `.plan-review.prev.md` snapshot — see § Prev snapshot); when supplied and readable, the browser highlights what changed since then. Omit it on the first launch; an unreadable `--prev` is non-fatal (the viewer renders without a diff). `--timeout <seconds>` caps the wait; omit it for serve.mjs's default (24h) and pass `--timeout 0` to disable the timer entirely (wait indefinitely).
- **`GET /api/plan`**: returns `{ "id": "<planId>", "markdown": "<raw plan markdown>", "lang": "<ja|en>", "prevMarkdown": "<raw markdown | null>" }`. The browser splits the markdown into sections, classifies them (Overview / Decisions / Design / Test plan / Risks / …), and renders the review surface; the server does **not** segment the plan or enumerate block ids. `prevMarkdown` is the `--prev` file's content (or `null` when `--prev` is absent / unreadable); when non-null the browser diffs current-vs-prev section by section, badges and auto-opens changed / new sections, collapses unchanged ones, and highlights changed blocks within changed sections.
- **stdout (in `--wait` mode)**: the **only** bytes written to stdout are the final submit JSON on one line — `JSON.parse`-able whole. All progress / errors go to stderr.
- **Submit JSON shape**: `{ "plan": "<planId>", "decision": "approve" | "revise", "submitted_at": "<iso8601>", "comments": [ { "block": "<id>", "excerpt": "<text>", "body": "<comment>" } ] }`. `comments` may be empty (an `approve` with no comments). `block` is a **browser-assigned semantic id**: `decision-<n>` for the n-th Decision card, or `<section-slug>::<n>` for the n-th commentable block (paragraph / list item / code block / …) within a section. `excerpt` is a short prefix of the commented block's text, for disambiguation.
- **Comments file**: serve.mjs also writes the same JSON to `<planId>.comments.json` next to the plan file, where `planId` is the plan file's basename with the trailing `.md` removed.
- **Exit codes**: `0` submit, `124` timeout (fires after the `--timeout` cap — default 24h; `--timeout 0` disables the timer so it never fires), `130` SIGINT / SIGTERM, `1` startup error (e.g. unreadable plan).

## Prev snapshot

To let the viewer show what changed across a revise round — so the user does not re-read the whole plan — the **previous** served-file content is preserved as `.claude/plans/<slug>.plan-review.prev.md` and passed to the next launch via `--prev`. Discipline: **whenever you are about to overwrite or edit the served file (`.plan-review.md`) and it already exists, first `cp` it to `.plan-review.prev.md`.** There are exactly two such sites (closed list):

- **(i)** in the **Decision mapping** step's `decision: "revise"` case, immediately before applying the block comments to the served file (the in-place `Edit`);
- **(ii)** in the **Write the served plan file** step, immediately before the `Write`, when the served file already exists — i.e. any non-first launch, including a `rewrite-approach` re-entry.

On the **first** launch the served file does not yet exist, so no prev is created and the launch omits `--prev` (no diff — there is nothing to diff against). The diff is a **guide to what changed**, not an authoritative diff: block moves and whitespace-only changes are not tracked precisely, and sections are matched by their title slug — a retitled section reads as one removed + one new, and reordering duplicate-titled sections can swap their match. `.plan-review.prev.md` is a workflow artifact covered by `SKILL.md` § Workflow artifacts (cross-step fixed exclusion) and removed in `SKILL.md` § Completion's cleanup.

## Procedure

1. **Resolve the serve.mjs path.** It lives at `<skill base directory>/scripts/plan-review/serve.mjs`, where `<skill base directory>` is the directory the harness reports as "Base directory for this skill" at this skill's invocation — the same mechanism `extract-rules` uses to run its bundled `scripts/extract_session_messages.mjs`. **Do not hardcode an absolute path** (the skill directory differs across the canonical tree, the bundle copy, and a user's plugin install — a base-relative path is the only portable form).

2. **Detect local-browser reachability.** Run `printenv CLAUDE_CODE_REMOTE`. If its value is `true` (the cloud-execution marker Claude Code on the Web sets — matching the official `[ "$CLAUDE_CODE_REMOTE" = "true" ]` guard), the local browser is unreachable: the agent runs in a remote headless sandbox with no port-forwarding and no display, so the served page never reaches the user. Return **`fallback`** to Step 4 immediately, without launching. Any other value — including unset / empty — means local execution (CLI / Remote Control); continue.

3. **Write the served plan file.** If `.claude/plans/<slug>.plan-review.md` **already exists** (a re-launch — e.g. a `rewrite-approach` re-entry), first `cp` it to `.claude/plans/<slug>.plan-review.prev.md` before overwriting (§ Prev snapshot, site (ii)). Then write the full plan body (the same content Step 4 sub-step 2 path (b) wrote to the canonical plan document `.claude/plans/<slug>.md`) to `.claude/plans/<slug>.plan-review.md`, where `<slug>` is the slug Step 4 sub-step 2 path (b) established. Write it **verbatim** — no block markers are inserted (the viewer derives block ids client-side from the plan's section structure, so markers are unnecessary). This **served file** is the gate's transient working copy, distinct from the canonical `.claude/plans/<slug>.md`; during the revise loop only the served file is updated, and `.claude/plans/<slug>.md` is written from the served file's latest content only when the gate exits via `approve` or `rewrite-approach` (steps 6–7).

4. **Launch the gate (background Bash).** Run `node "<skill base directory>/scripts/plan-review/serve.mjs" --plan ".claude/plans/<slug>.plan-review.md" --lang <resolved-language> --wait` as a **background Bash** command (`run_in_background: true`), substituting the workflow's resolved `language` for `<resolved-language>`, and emit a Progress Visibility status line in the same turn (e.g. "Launching the visual plan-review gate in your browser…"). **When `.claude/plans/<slug>.plan-review.prev.md` exists** (a re-launch where a prev snapshot was taken — § Prev snapshot), append `--prev ".claude/plans/<slug>.plan-review.prev.md"` so the viewer shows the diff; omit it on the first launch. Do **not** pass `--timeout`: the gate uses serve.mjs's 24h default — long enough to never fire during a real review, but still an eventual fallback if the gate is abandoned. Waiting costs no tokens (the caller blocks on a harness-tracked background process), so a long wait has no cost. Pass `--timeout 0` only if truly no timeout (wait indefinitely) is wanted. Notes:
   - This is background **Bash**, **not** the `Agent` tool (see `SKILL.md` § Configuration's `Agent` tool usage bullet — the gate is not a subagent dispatch and does not count against the fixed `Agent` dispatch sites).
   - Background dispatch is **required** here: the gate's wait can run up to 24h (or indefinitely with `--timeout 0`), which a foreground Bash call (~10-min limit) could not survive. The harness re-invokes the agent when the background process exits. Waiting for that exit is a **harness-tracked boundary**, not a "type continue" pause (see `SKILL.md` § No-Stall Principle) — proceed on the exit notification.

5. **Read the decision.** When the background process exits, read its captured stdout. In `--wait` mode the only stdout content is the submit JSON (one line) — parse it. The decision comes from this stdout; the on-disk comments file (`.claude/plans/<slug>.plan-review.comments.json`, per § serve.mjs contract's derivation) is a durable side record that is **not** read here. The served file, the comments file, and the prev snapshot (`.plan-review.prev.md`) are workflow artifacts covered by `SKILL.md` § Workflow artifacts (cross-step fixed exclusion) — excluded from Step 10 commits automatically (no `.gitignore` entry needed).

6. **Decision mapping.** Evaluate the parsed result in order (first match wins):
   - **Non-zero exit** (`124` timeout — fires only after the cap elapses (default 24h), never with `--timeout 0`; `130` SIGINT / SIGTERM; `1` startup error) — or no parseable result obtained — → return **`fallback`** to Step 4 (e.g. the gate was cancelled, or a closed browser left it waiting until the cap elapsed; Step 4's no-Plan-Mode chat-approval path lets them approve / request changes via a chat reply — no modal, since this path runs outside Plan Mode).
   - **`decision: "approve"`** → maps to Step 4's `accept`. Any accompanying `comments` are **advisory only** — record them but do not block on them (the user chose approve). Write the served file's latest content to the canonical `.claude/plans/<slug>.md`, then return **`approve`** to Step 4 — the browser submit is itself the final confirmation, so Step 4 proceeds directly to implementation (**no `ExitPlanMode`**, since this path runs outside Plan Mode).
   - **`decision: "revise"`** → **first** `cp` the served file to `.claude/plans/<slug>.plan-review.prev.md` (§ Prev snapshot, site (i)) — capturing what the user just reviewed so the next launch can diff against it — **then** apply the block comments **directly to the served file** (`.claude/plans/<slug>.plan-review.md` — the only plan artifact during the loop). For each comment, map its `block` id to the corresponding part of the plan and apply the requested change autonomously (same discipline as Step 3 finding application — the user's block comments are the review):
     - **Block-id resolution**: `decision-<n>` → the n-th Decision item; `<section-slug>::<n>` → the n-th commentable block within that section. **Resolve by the `excerpt` first**: it is a whitespace-normalized, ≤40-char prefix of the commented block's text, so locate the block whose leading text matches it after the same normalization (collapse runs of whitespace, then compare the prefix) rather than requiring an exact substring match. Treat the numeric index (`<n>`) as a secondary, advisory hint only — the browser derives it from a client-side segmentation in `public/index.html` that an agent editing the served markdown cannot reproduce, so never rely on the index alone. **Block ids are per-render ephemeral by design**: every `revise` submit's comments are applied to (and thereby consumed from) the served file in this same step, and the next render assigns fresh ids — so an id never needs to remain stable across renders. This is **not** a regression from the prior marker scheme; the consume-on-each-submit model removes the need for cross-render id stability.
     - **"Switch to alternative" comments**: a Decision card's toggle emits a comment on `decision-<n>` whose `body` is the fixed switch instruction (`この Decision を Alternative に切り替える` for `lang: ja` / `Switch this decision to its Alternative` for `lang: en`, optionally followed by free-text). Treat it as a **bounded Recommendation/Alternative swap** on that Decision: swap the item's Recommendation and Alternative in the served file and propagate the dependent updates (Approach / Risks / Scope / Test-plan sweep) exactly as the text-path swap-decisions bucket does (see `SKILL.md` Step 4's swap-decisions handling). When the `body` carries free-text beyond the fixed switch sentence (the portion after that sentence), apply **both legs**: perform the swap first, then apply that trailing portion as an ordinary revise instruction on the same Decision — the swap is not a license to drop the rest of the comment. The browser toggle is itself the confirmation gesture, so the text-path read-back sub-step is **intentionally omitted** here.
     This comment-application happens for **every** `revise` submit, before and regardless of the branch below. Then branch on the nature of the (already-applied) changes:
     - **Approach-level material change** (a new overall approach, a scope change — same vocabulary as Step 4's `rewrite-approach` bucket: "Approach / Design / Scope-level material change"): do **not** loop the gate. Write the served file's latest content to the canonical `.claude/plans/<slug>.md` (so the re-review sees the current plan) and return **`rewrite-approach`** to Step 4, which adds Step 3-(N_plan+1), re-reviews the plan, and re-enters Step 4 from sub-step 1 — keeping review-quality symmetric with the default path for material changes. (On re-entry the visual / chat branch is re-evaluated; if still visual, the gate re-runs on the re-reviewed plan.)
     - **Localized edits** (wording, a single Decision tweak, a bounded Recommendation/Alternative swap, a bounded section edit): **re-launch the gate** (loop back to step 4) — the edits already landed in the served file above. The human reviewing in the browser is the review pass, so Step 3 is not re-run for localized revise. The loop is human-paced — each iteration blocks on a fresh browser submit, so no machine iteration cap is needed (unlike the Step 3 / Step 8 / Step 10 bounded loops). A closed browser does **not** by itself end the gate (the server keeps waiting); the gate exits via `fallback` only on the timeout cap (default 24h, never with `--timeout 0`), a SIGINT / SIGTERM, or a startup error (the non-zero-exit row above).

7. **Fallback contract.** Return **`fallback`** to Step 4 on any of: local browser unreachable (`CLAUDE_CODE_REMOTE="true"`, step 2), non-zero serve.mjs exit, a startup / launch failure, or the launch Bash call being blocked / yielding no result. Step 4 then continues with its **no-Plan-Mode chat-approval path** (sub-step 2 path (b)'s fallback branch): the chat-rendered condensed plan → preamble → guidance line → a pointer to `.claude/plans/<slug>.md` → wait for the user's chat reply — **no `ExitPlanMode`**, since this whole path runs outside Plan Mode. A `fallback` never silently stalls the workflow — Step 4 always has the chat-approval path to fall through to.

The three return values to Step 4 are **`approve`**, **`rewrite-approach`**, and **`fallback`** (routing defined in § Decision mapping above).
