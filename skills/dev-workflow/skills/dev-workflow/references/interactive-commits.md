# Interactive Commits (Step 10) ‚Äî Procedure

Single canonical home for the Step 10 (Interactive Commits) procedure body ‚Äî `SKILL.md` ¬ß Step 10 keeps only the entry conditions, the `landed_count` cross-step contract, the **Approval token closed list**, and the **Localized summary tokens**, and points here for the procedure; do not duplicate this content back into `SKILL.md`. Only two unqualified `¬ß` references below resolve to `SKILL.md`, where those definitions stay: `¬ß Approval token closed list` and `¬ß Localized summary tokens`. All other `¬ß` references resolve within this file.

If the current plan explicitly deferred bookkeeping edits to the commit stage ‚Äî for example, CHANGELOG entries or version-bump lines whose content was deferred until the implementation is confirmed complete ‚Äî apply those edits now, before Procedure 1 collects the working tree, so they are captured in the commit grouping. Use the plan's deferral language (e.g. "will add the CHANGELOG entry at commit time", "version bump deferred to Step 10") as the trigger. Apply these deferred edits autonomously, with no user gate before Procedure 1. If any edit call fails, stop Step 10 and report the failure to the user.

**Procedure**:

1. **Collect changes**: `<base-commit>` referenced throughout this Procedure is the value recorded by Step 2's opening sub-step (`git rev-parse HEAD` at workflow start ‚Äî see `SKILL.md` ¬ß Step 2: Create Plan's first numbered item); Step 10 reuses that captured value verbatim, never re-resolves it. Run `git status --porcelain=v1 --untracked-files=all -z` once (the `=v1` form pins format; `--untracked-files=all` overrides any user-side `status.showUntrackedFiles=no` config; `-z` emits NUL-separated entries with no C-style quoting so filenames containing spaces, quotes, or non-ASCII characters are recoverable verbatim). Parse the NUL-separated output: entries prefixed `??` are untracked, others are tracked changes. If the output is empty, mark `Step 10: Interactive Commits` as `completed` and proceed to Step 11 (nothing to commit since base-commit). Otherwise keep the parsed file list in memory as the canonical roster. Then run `git diff <base-commit>` once to capture the per-file diff for **tracked** files (the single `git diff` output is consumed per-file at `Per-commit loop`'s `Present` step by slicing on `diff --git a/<path> b/<path>` boundary markers ‚Äî do not issue separate per-file `git diff <base-commit> -- "<path>"` invocations); for **untracked** files the diff is empty, so for each untracked path also `Read` its file contents and treat that as the "diff" to present at `Per-commit loop`'s `Present` step (since `git diff` omits untracked paths by design)
2. **Deduce commit style**: run `git log -n 10 --format=%s` and infer the project's conventional commit style (presence of type prefix, imperative mood, etc.) ‚Äî used as a runtime hint when drafting subjects, never hard-coded
3. **Propose commit plan**: Before grouping, run the plan-scope cross-reference below, then group changes into 1+ logical commits, each `{subject, body (optional, may be empty), files}`. Group by cohesion (concerns that differ ‚Üí separate commits); typical case is a single commit.

   **Plan-scope cross-reference**: scan the approved plan for explicit "no-change" declarations ‚Äî files, sections, or regions the plan stated would not be modified (phrasings like `"no changes"`, `"leave as-is"`, `"unchanged"`, `"not modified"`, `"read-only"`, or equivalent intent). Cross-reference against the collected change set from Procedure 1. If any collected change touches a declared no-change area, surface those paths as **unplanned changes** at the top of the commit-plan presentation ‚Äî above the numbered commit list ‚Äî naming each affected path and quoting the relevant plan text verbatim. Do not silently include unplanned changes in the proposed grouping; the user resolves them via the existing commit-plan approval gate's `adjust` (move to a separate commit with explanation, or exclude by omitting from the commit pathspec (leave the path out of the `-- <path>` list in the `d. Commit` sub-step) ‚Äî do **not** use `git checkout HEAD -- <path>` to exclude, as that destroys existing uncommitted changes to the file) or `cancel` branches. When no declared no-change areas exist in the plan, or no collected change overlaps with them, skip this cross-reference and proceed directly to the grouping.

   Present the plan to the user as a numbered list (subject + file list per commit) and wait at the **commit-plan approval gate**. Approval is judged per ¬ß Approval token closed list.
4. **Per-commit loop** (`landed_count` was initialized to `0` at Step 10 entry in `SKILL.md`; no re-initialization is needed here ‚Äî the counter is incremented only at sub-step `d. Commit` on a zero-exit `git commit`): process each commit in order:
   - **a. Present**: Before rendering the four presentation elements, perform one upfront size measurement: sum the character count across all per-file diff slices for this commit's files (tracked files: their portion of the Procedure 1 `git diff <base-commit>` output; untracked files: their `Read` contents). Decide the diff rendering mode once ‚Äî three modes in order, first match wins ‚Äî and hold that decision for every file in the Diff element; never re-evaluate mid-presentation:
     - **verbatim**: two gates evaluated in order, short-circuiting on the first that passes. **First** the line-count gate ‚Äî total line count ‚â§ `diff_verbatim_line_threshold` (default 100 lines; set in `.claude/dev-workflow.local.md` to override ‚Äî use `99999999` to always pass the line-count gate); if it passes, the mode is verbatim and the char-count gate is **not** consulted. **Otherwise** the char-count gate ‚Äî total character count ‚â§ `diff_verbatim_threshold` (default 4000 chars; set in `.claude/dev-workflow.local.md` to override ‚Äî use `99999999` to always render verbatim); passing it also selects verbatim. The line-count gate is checked first so long-line prose diffs with a small line count are not downgraded by the char-count test
     - **condensed**: total character count ‚â§ `diff_condensed_threshold` (default 20000 chars; set in `.claude/dev-workflow.local.md` to override ‚Äî use `99999999` to always render condensed, never skeleton) ‚Äî render first 100 lines per file with a `‚Ä¶ [N lines omitted]` trailer, where N is the remaining line count for that file
     - **skeleton**: total character count exceeds `diff_condensed_threshold` ‚Äî render first 20 lines per file followed by `‚Ä¶ [N lines omitted ‚Äî run git diff <base-commit> -- "<path>" to view the full diff for this file]`, where N is the remaining line count; all files are included (none fully skipped) so the user sees each file's header and opening context regardless of diff size

     Then show the four presentation elements as a **closed list** ‚Äî Subject / Body / Files / Diff ‚Äî each rendered in its own block as the user's approval basis:
     - **Subject**: render in a dedicated fenced code block (single line, no surrounding prose).
     - **Body**: render in a dedicated fenced code block. When the commit has no body, render the placeholder `(no body)` inside the block ‚Äî a prose statement like "body included" / "no body needed" without an actual rendered block is **forbidden**, because the user's approval must be anchored to the material content itself, not a promise of it.
     - **Files**: render as an explicit pathspec list (one path per line, prefixed with `- `) ‚Äî the same paths that will be passed to `c. Stage`'s `git add -- <paths>` invocation.
     - **Diff**: render the actual per-file hunks using the mode chosen in the upfront measurement. For tracked files, use the per-file portion of the diff captured at Procedure 1; for **untracked** files, use the full file contents (also captured at Procedure 1) shown as a "new file" hunk ‚Äî make the untracked status explicit in the per-file label so the user can distinguish "added file" from "modified file" before approving.

     Prose-only summaries such as "body Âê´„ÇÅ„ÄÅdiff full preview" / "subject and body included" without actually rendering the corresponding blocks are forbidden ‚Äî they leave the user without the material content needed to judge approve / adjust / cancel and trigger an immediate re-render request
   - **b. Per-commit accept gate**: wait for the user. Categorize the response per ¬ß Approval token closed list
   - **c. Stage**: run `git add -- "<file-1>" "<file-2>" ...` in a single invocation with one explicit pathspec per file (verbatim filenames recovered from Procedure 1's `-z` output ‚Äî never the C-quoted form). The `--` separator + double-quoting handle spaces, quotes, and non-ASCII characters; `-A` and other bulk forms are forbidden because they may stage unrelated drift
   - **d. Commit**: prefer a HEREDOC even for one-line subjects so multi-line bodies share the same form. Always include the same pathspec that `c. Stage` used ‚Äî this prevents any auto-staging hook that fires between Stage and Commit from expanding the commit's scope. `git commit -m "<subject>" -- "<path-1>" "<path-2>" ...` for subject-only; for subject + body use:
     ```bash
     git commit -F - -- "<path-1>" "<path-2>" ... <<'EOF'
     <subject>

     <body>
     EOF
     ```
     On zero exit, **increment `landed_count` by 1**.
   - **e. Non-zero-exit retry (commit-attempt failure)**: if `git commit` exits non-zero, retry once after a 1‚Äì2 second sleep. On the second non-zero exit, render `commit-failed (<reason>)` where `<reason>` is the last non-empty line of stderr truncated to ‚â§ 80 characters; if stderr is empty or whitespace-only, render `commit-failed (no stderr)`. Then **stop Step 10**. **Do not increment `landed_count`**. Do not auto-recover (no force, no reset, no rebase, no amend). For pre-commit hook rejections, instruct the user to fix the hook out-of-band and restart `/dev-workflow` in a new session
5. **Post-commit auto-modify cycle bound**: after a zero-exit commit, re-run `git status --porcelain=v1 --untracked-files=all -z`. If new uncommitted changes appeared (a pre-commit hook auto-fixed the tree), ask the user a fold-or-defer question with explicit semantics ‚Äî **`fold` (= staple the hook's edits into the just-landed commit)** or **`defer` (= leave them in the working tree for a later commit-plan iteration)**. Map the response per the **dedicated 5-branch ‚Üí `fold` / `defer` / `cancel` / re-present-as-`adjust` classifier below** ‚Äî the 5 input branches extend ¬ß Approval token closed list's 4 buckets (`accept` / `adjust` / `cancel` / `NOT approval`) with a 5th `defer-direction` branch specific to this gate, mapping to 4 distinct dispositions (explicit per-branch mapping ‚Äî no `everything else` catch-all, so ambiguous / interrogative responses cannot silently land in `defer`):
   - **accept** (affirmation of folding ‚Äî "fold" / "amend" / "Áèæ„Ç≥„Éü„ÉÉ„Éà„Å´Âê´„ÇÅ„Çã" or any semantic equivalent for incorporating the hook's edits) ‚Üí `fold`
   - **adjust** (specific revision request ‚Äî "defer just foo.ts but fold bar.ts" / "rename the subject before folding" / any concrete change demand that is neither a clean fold nor a clean defer) ‚Üí re-present the gate via ¬ß Mid-loop adjust branch **f** `Ambiguous / un-classifiable adjust request` (clarifier); do **NOT** silently route to `defer`
   - **cancel** / **stop** ‚Üí `Mid-loop cancel` per its canonical route
   - **NOT approval** (interrogative or non-committal ‚Äî "look good?" / "„Å©„ÅÜÔºü" / "„Åì„Çå„Åß„ÅÑ„ÅÑÔºü") ‚Üí treat as `adjust` per ¬ß Approval token closed list and re-present the gate; do **NOT** silently route to `defer`
   - Otherwise ‚Äî an unambiguous defer-direction response ("defer" / "Âæå„Åß" / "leave it" / "skip" / "later" or any semantic equivalent for postponing the hook's edits) ‚Üí `defer`

   On `fold`: `git add -- "<file-1>" "<file-2>" ...` with **exactly the files reported by this just-issued post-commit porcelain re-run** (the hook-modified set ‚Äî not the original commit's file list, not the Procedure 1 roster); without this stage step `--amend` would silently keep the pre-hook tree. Then `git commit --amend --no-edit` (or `git commit --amend -F -` with the original subject + body) to incorporate them. The amend re-commits the same logical commit ‚Äî `landed_count` is **not** re-incremented. On `defer`: leave the working tree as the hook left it; the next iteration of `Per-commit loop` (or a subsequent plan re-render via the `Mid-loop adjust` file-regrouping branch) will surface them. Re-commit is allowed at most once per logical commit; if a second commit attempt on the **same logical commit** also triggers auto-modify, abort Step 10 with `pre-commit hook unstable (commit <#k>: hook re-modified <files>)` where `<files>` is the porcelain-reported file list at the time of abort. Emit the partial-completion summary (same retry-once + stop policy as the `Per-commit loop`'s `Non-zero-exit retry (commit-attempt failure)` step). Surfacing the file list gives the user actionable context ‚Äî they typically need to disable / fix the offending hook out-of-band before resuming
6. **Landed-commits invariant**: once a commit has zero-exit landed, it is **immutable**. The `adjust` flows below apply only to the un-landed commits starting from `#landed_count + 1`. Landed commits are shown in the numbered list as history for context but cannot be re-edited from inside Step 10
7. **Mid-loop adjust ‚Äî closed-list branches** (treat any user response categorized as `adjust` per ¬ß Approval token closed list).

   **Gate-of-origin tracking**: every time Step 10 dispatches one of its three gates ‚Äî commit-plan approval gate (Procedure 3), per-commit accept gate (Procedure 4 sub-step `b`), or fold-or-defer gate (Procedure 5) ‚Äî set `gate_origin` to that gate's identifier before waiting for the user response. Branches **e**, **f**, and **g** below read `gate_origin` to know which gate to re-enter; without this tracking, an adjust-request raised at the fold-or-defer gate could silently fall back to the per-commit accept gate (or vice versa), losing the gate-specific framing (e.g. the in-flight just-landed commit's identity for `fold` decisions).

   - **a. Subject / body change only** on the in-progress un-landed commit: update the field, then jump back to `Per-commit loop`'s `Present` step for the same commit (do not move the per-commit pointer)
   - **b. File regrouping** (merge / split / reorder / add / remove ‚Äî **including splits or merges that change the un-landed commit count**, since both operations rebuild the un-landed portion and re-render): rebuild the un-landed portion of the plan, **re-render the numbered list in full** (with the landed prefix shown as history), and re-enter the commit-plan approval gate. On re-approval, the per-commit pointer resets to `#landed_count + 1` and execution resumes at `Per-commit loop`'s `Present` step for that pointer
   - **c. Un-landed portion drops to 0 commits** (the user's adjust removed everything that was still pending): mark `Step 10: Interactive Commits` as `completed` and proceed to Step 11 ‚Äî this is a normal completion, not a cancel
   - **d. Merge absorbs the un-landed remainder into an already-landed commit** ‚Üí un-landed = 0: same disposition as the **Un-landed portion drops to 0 commits** branch above (normal completion)
   - **e. Adjust request targets an already-landed commit** (e.g., "change the subject of commit #1 (already landed)", "move file X from commit #1 to commit #3"): reject the request, surface the landed-commits invariant explanation (`landed commits are immutable inside Step 10 ‚Äî exit and use git rebase -i / commit --amend outside the workflow if needed`), and re-enter the gate identified by `gate_origin`
   - **f. Ambiguous / un-classifiable adjust request**: ask the user a clarifying question and re-enter the gate identified by `gate_origin` (do not fall through silently)
   - **g. Actual-code changes made during the gate** (the user's adjust response indicates they edited a file in the commit's proposed set since Procedure 1's capture): distinguish metadata-only edits (comment rewording, documentation text, non-functional spacing) from actual-code edits (logic, function signatures, types, control flow, or any change that could affect behavior). For metadata-only edits: re-capture the file's updated content, update the commit description if appropriate, and re-enter the gate identified by `gate_origin`. For actual-code edits: surface the updated diff to the user and require an explicit choice ‚Äî **(1) continue without re-verification** (user acknowledges the change is a targeted fix consistent with the plan and no re-test is needed; proceed to `c. Stage` with updated file contents) or **(2) stop for re-verification** (stop Step 10, leave working-tree changes as-is, and instruct the user to re-run verification from Step 7 before continuing)
8. **Mid-loop cancel**: a response categorized as `cancel` per ¬ß Approval token closed list stops Step 10 with all landed commits preserved and all un-landed changes left in the working tree as-is. **The workflow does not revert** ‚Äî the user is free to `git reset` / `git stash` / continue manually. Record the partial state per ¬ß Localized summary tokens
9. After all commits land, or the user cancels (see `Mid-loop cancel`), or the `Mid-loop adjust` un-landed-drops-to-zero / merge-absorbs-into-landed branches complete the un-landed work, mark `Step 10: Interactive Commits` as `completed` and proceed to Step 11
