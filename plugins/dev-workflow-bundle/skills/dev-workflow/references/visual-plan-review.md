# Visual Plan-Review Gate (Step 4)

Deep reference for Step 4's **visual gate path**. Read and follow this top to bottom only when Step 4 sub-step 2 has selected the visual gate — i.e. `visual_plan_review` is enabled (§ Configuration) and the local browser is reachable. This file is the single canonical home for the gate procedure; Step 4 delegates here rather than restating it.

The gate replaces Step 4's text approval presentation with a browser-based, block-level plan review. It returns one of three outcomes to Step 4: **`approve`**, **`rewrite-approach`**, or **`fallback`** (defined in § Decision mapping and § Fallback contract below).

## serve.mjs contract

The viewer (`scripts/plan-review/serve.mjs`) ships with this skill; this gate only invokes it. Its contract:

- **Invocation**: `node <serve.mjs> --plan <path> --wait` — serves the Markdown plan at `<path>` on a random `127.0.0.1` port, opens the local default browser, and blocks until the user submits.
- **stdout (in `--wait` mode)**: the **only** bytes written to stdout are the final submit JSON on one line — `JSON.parse`-able whole. All progress / errors go to stderr.
- **Submit JSON shape**: `{ "plan": "<planId>", "decision": "approve" | "revise", "submitted_at": "<iso8601>", "comments": [ { "block": "<id>", "excerpt": "<text>", "body": "<comment>" } ] }`. `comments` may be empty (an `approve` with no comments).
- **Comments file**: serve.mjs also writes the same JSON to `<planId>.comments.json` next to the plan file, where `planId` is the plan file's basename with the trailing `.md` removed.
- **Exit codes**: `0` submit, `124` timeout (default 1800s / 30 min), `130` SIGINT / SIGTERM, `1` startup error (e.g. unreadable plan).
- **Block segmentation**: explicit `<!-- block:bN -->` markers are authoritative when present; otherwise serve.mjs falls back to deterministic heading-based numbering. The gate writes markers (step 3 below) so block ids stay stable across revise loops.

## Procedure

1. **Resolve the serve.mjs path.** It lives at `<skill base directory>/scripts/plan-review/serve.mjs`, where `<skill base directory>` is the directory the harness reports as "Base directory for this skill" at this skill's invocation — the same mechanism `extract-rules` uses to run its bundled `scripts/extract_session_messages.mjs`. **Do not hardcode an absolute path** (the skill directory differs across the canonical tree, the bundle copy, and a user's plugin install — a base-relative path is the only portable form).

2. **Detect local-browser reachability.** Run `printenv CLAUDE_CODE_REMOTE`. If its value is `true` (the cloud-execution marker Claude Code on the Web sets — matching the official `[ "$CLAUDE_CODE_REMOTE" = "true" ]` guard), the local browser is unreachable: the agent runs in a remote headless sandbox with no port-forwarding and no display, so the served page never reaches the user. Return **`fallback`** to Step 4 immediately, without launching. Any other value — including unset / empty — means local execution (CLI / Remote Control); continue.

3. **Write the served plan file (with block markers).** Write the full plan body (the same content the text path writes to the Plan Mode plan file) to `.claude/plans/<slug>.plan-review.md`, where `<slug>` is the plan slug established in Step 1. Insert a `<!-- block:bN -->` marker on its own line immediately before each top-level section so block ids are stable across revise loops (HTML comments are invisible in both the browser render and the `ExitPlanMode` modal). This **served file** is distinct from the Plan Mode plan file (which the `ExitPlanMode` modal renders); during the revise loop only the served file is updated, and the Plan Mode plan file is written from the served file's latest content only when the gate exits via `approve` or `rewrite-approach` (steps 6–7).

4. **Launch the gate (background Bash).** Run `node "<skill base directory>/scripts/plan-review/serve.mjs" --plan ".claude/plans/<slug>.plan-review.md" --wait` as a **background Bash** command (`run_in_background: true`), and emit a Progress Visibility status line in the same turn (e.g. "Launching the visual plan-review gate in your browser…"). Do not pass `--timeout` — serve.mjs's default (1800s) applies. Notes:
   - This is background **Bash**, **not** the `Agent` tool (see `SKILL.md` § Configuration's `Agent` tool usage bullet — the gate does not count against the three subagent-dispatch steps).
   - Background dispatch is required because the foreground Bash tool's max timeout (~10 min) is shorter than serve.mjs's 30-min wait; the harness re-invokes the agent when the background process exits. Waiting for that exit is a **harness-tracked boundary**, not a "type continue" pause (see `SKILL.md` § No-Stall Principle) — proceed on the exit notification.

5. **Read the decision.** When the background process exits, read its captured stdout. In `--wait` mode the only stdout content is the submit JSON (one line) — parse it. The decision comes from this stdout; the on-disk comments file (`.claude/plans/<slug>.plan-review.comments.json`, per § serve.mjs contract's derivation) is a durable side record that is **not** read here. Both the served file and the comments file are workflow artifacts covered by `SKILL.md` § Workflow artifacts (cross-step fixed exclusion) — excluded from Step 10 commits automatically (no `.gitignore` entry needed).

6. **Decision mapping.** Evaluate the parsed result in order (first match wins):
   - **Non-zero exit** (`124` timeout, `130` SIGINT / SIGTERM, `1` startup error) — or no parseable result obtained — → return **`fallback`** to Step 4 (the user may have simply closed the browser; the proven text path lets them approve / withdraw via the modal).
   - **`decision: "approve"`** → maps to Step 4's `accept`. Any accompanying `comments` are **advisory only** — record them but do not block on them (the user chose approve). Write the served file's latest content to the Plan Mode plan file, then return **`approve`** to Step 4 (Step 4 calls `ExitPlanMode`).
   - **`decision: "revise"`** → apply the block comments **directly to the served file** (`.claude/plans/<slug>.plan-review.md` — the only plan artifact during the loop): for each comment, map its `block` id to the corresponding section and apply the requested change autonomously (same discipline as Step 3 finding application — the user's block comments are the review). This comment-application happens for **every** `revise` submit, before and regardless of the branch below. Then branch on the nature of the (already-applied) changes:
     - **Approach-level material change** (a new overall approach, a scope change — same vocabulary as Step 4's `rewrite-approach` bucket: "Approach / Design / Scope-level material change"): do **not** loop the gate. Write the served file's latest content to the Plan Mode plan file (so the modal / re-review sees the current plan) and return **`rewrite-approach`** to Step 4, which adds Step 3-(N_plan+1), re-reviews the plan, and re-enters Step 4 from sub-step 1 — keeping review-quality symmetric with the text path for material changes. (On re-entry the visual / text branch is re-evaluated; if still visual, the gate re-runs on the re-reviewed plan.)
     - **Localized edits** (wording, a single Decision tweak, a bounded section edit): **re-launch the gate** (loop back to step 4) — the edits already landed in the served file above. The human reviewing in the browser is the review pass, so Step 3 is not re-run for localized revise. The loop is human-paced — each iteration blocks on a fresh browser submit, so no machine iteration cap is needed (unlike the Step 3 / Step 8 / Step 10 bounded loops); a closed browser or timeout exits via `fallback` (the non-zero-exit row above).

7. **Fallback contract.** Return **`fallback`** to Step 4 on any of: local browser unreachable (`CLAUDE_CODE_REMOTE="true"`, step 2), non-zero serve.mjs exit, a startup / launch failure, or the launch Bash call being blocked / yielding no result. Step 4 then continues with its text path (chat-rendered condensed plan → preamble → guidance line → `ExitPlanMode`). A `fallback` never silently stalls the workflow — Step 4 always has the text path to fall through to.

The three return values to Step 4 are **`approve`**, **`rewrite-approach`**, and **`fallback`** (routing defined in § Decision mapping above).
